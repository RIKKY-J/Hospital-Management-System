# 🏥 City Hospital Management System

A lightweight, object-oriented full-stack web application for hospital management and automated appointment scheduling. This application is built with a **Node.js/Express** backend, a **file-based JSON database**, a **vanilla HTML/CSS/JS frontend**, and is fully tested using **Selenium WebDriver**, **Mocha**, and **Chai**, with rich test reports generated by **Allure Quality Reporting**.

---

## 🚀 Key Features

*   **OOP-Driven Backend Architecture**: Domain models (`Doctor`, `Patient`, `Appointment`, `Staff`) implemented as clean JavaScript classes with strict separation of concerns.
*   **JSON-Backed Storage**: A custom, lightweight persistence layer (`Database.js`) that reads and updates a local JSON file (`db.json`), ensuring state preservation across runs.
*   **Dynamic Client Portal**: A responsive frontend for patients to book appointments, view available doctors, and check schedules in real time.
*   **Staff Login & Dashboards**: Dedicated portal for hospital staff with role-based routing (Admin, Doctor, Nurse, Staff).
*   **Interactive Admin Controls**: Authorized admin dashboard featuring statistics, list views for patients and doctors, and appointment cancellation capabilities.
*   **Automated E2E Testing**: Comprehensive Selenium WebDriver browser automation suite to test the critical user pathways (login, booking, admin actions).
*   **Allure Quality Reports**: Visual reporting framework integration to generate and inspect test execution details, failure logs, and step timelines.

## 🔌 API Documentation

The server exposes a clean REST API on `http://localhost:3000` to interact with the database:

### Core Routes

| Method | Endpoint | Request Body | Description |
| :--- | :--- | :--- | :--- |
| **GET** | `/api/doctors` | *None* | Retrieve all doctors and their specialities |
| **GET** | `/api/users` | *None* | Retrieve list of registered patients |
| **GET** | `/api/appointments` | *None* | Get list of all booked appointments |
| **POST** | `/api/appointments` | `{ name, email, phone, doctorId, date, time }` | Schedule a new appointment. Registers a new patient if they do not exist. |
| **DELETE** | `/api/appointments/:id`| *None* | Cancel/delete an appointment by unique numeric ID |
| **POST** | `/api/login` | `{ username, password }` | Authenticate staff credentials, return custom session token and user role |
| **GET** | `/api/admin/all` | *None* | Fetch the complete database file structure (Doctors, Users, Appointments, Staff) |
| **GET** | `/api/admin/stats` | *None* | Retrieve high-level statistics count for dashboards |

## ⚙️ Installation & Running

### Prerequisites
Make sure you have [Node.js](https://nodejs.org/) and the [Google Chrome Browser](https://www.google.com/chrome/) installed.

### Setup Steps
1. **Clone the Repository**:
   ```bash
   git clone https://github.com/RIKKY-J/Hospital-Management-System.git
   cd Hospital-Management-System
   ```
2. **Install Dependencies**:
   ```bash
   npm install
   ```
3. **Start the Web Server**:
   ```bash
   npm start
   ```
4. **Access the Application**:
   Open your browser and navigate to:
   *   **Main Site**: `http://localhost:3000`
   *   **Booking Portal**: `http://localhost:3000/book.html`
   *   **Staff Login**: `http://localhost:3000/staff.html`

## 🧪 Running the E2E Test Suite

The automated tests are configured to run locally via Chrome browser automation using **Selenium WebDriver**.

1. Ensure the Express server is running in a terminal:
   ```bash
   npm start
   ```
2. Run the test suite:
   ```bash
   npm test
   ```
   *This command runs the Mocha test suite inside a Chrome browser sandbox and stores results in the `./allure-results` folder.*

### 📊 Generating Allure Reports
To compile and view the test reports in an interactive browser window:
```bash
npm run report
```
*This command cleans up previous results, compiles test runs, creates the static `allure-report` folder, and automatically spins up a local server to view the report dashboard.*

## 🏛️ OOP Design Pattern Summary

This project implements a solid **Domain-Driven OOP Architecture**:

1.  **`Database` Class**: Encapsulates all read/write file transactions. It abstracts low-level file manipulations, preventing data corruption and implementing ID generation.
2.  **`HospitalSystem` Controller**: Acts as a **Facade**. The server router makes calls exclusively through `HospitalSystem` methods. It orchestrates logins, formats statistic packages, and coordinates links between `Patient` and `Appointment` instances.
3.  **`Patient` Class**: Tracks personal metrics and links internal lists of historical or pending `Appointment` classes.
4.  **`Staff` Class**: Encapsulates role privileges (e.g. checking `isAdmin()`), and records `lastLogin` timestamp.
5.  **`Appointment` Class**: Includes status checks (e.g., booked, checked-in, completed) and helper functions like `isUpcoming()` which performs date comparisons.
